②Docker · Multi-stage Build

3단계 빌드로 최소 이미지

최종 이미지에는 빌드 도구가 포함되지 않도록 stage를 분리.

STAGE 1

deps

의존성 캐시 레이어

FROM node:24-alpine COPY package*.json prisma ./ RUN npm ci
STAGE 2

build

Next.js 빌드 + Prisma generate

FROM node:24-alpine COPY --from=deps node_modules RUN prisma generate RUN next build
STAGE 3

runtime

non-root 실행 + healthcheck

USER app EXPOSE 3000 HEALTHCHECK /api/health CMD ["node","server.js"]
Multi-stage 빌드 도구 미포함 → 최소 이미지
Non-root user USER app — 보안 강화
HEALTHCHECK /api/health — 컨테이너 상태 검증
docker-compose app + Postgres + Nginx 한 번에 실행
BuildKit cache CI에서 GHA 캐시 재사용
CI smoke build 모든 PR에서 Dockerfile 깨짐 방지

③Nginx · Reverse Proxy

단일 도메인 → 3-way 라우팅

nginx/nginx.conf의 location 블록으로 정적 자산 · WebSocket · API를 분기해 upstream app:3000 으로 프록시.

/_next/static/ 정적 자산 
location /_next/static/ {
  proxy_pass http://app;
  proxy_set_header Host $host;
  expires 1y;
  add_header Cache-Control "public, immutable";
  access_log off;
}
해시 파일명이라 1년 immutable로 강하게 캐싱. 로그도 끄고 비용/소음 모두 절약.
/socket.io/ WebSocket 
location /socket.io/ {
  proxy_pass http://app;
  proxy_http_version 1.1;
  proxy_set_header Upgrade $http_upgrade;
  proxy_set_header Connection "upgrade";
  proxy_read_timeout 86400;
  proxy_buffering off;
}
가장 잘 깨지는 구간. Upgrade/Connection 헤더가 없으면 즉시 HTTP로 다운그레이드돼 실시간이 멈춤.
/ API · 페이지 
location / {
  proxy_pass http://app;
  proxy_http_version 1.1;
  proxy_set_header Host $host;
  proxy_set_header X-Real-IP $remote_addr;
  proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
  proxy_read_timeout 60s;
}
기본 캐치-올. X-Forwarded-*로 클라이언트 IP/scheme을 Next.js까지 정확히 전달.

④Web Storage · 3종 동시 활용

localStorage · sessionStorage · IndexedDB

요건은 셋 중 하나였지만, 셋 모두 서로 다른 목적으로 활용.

localStorage

영속적 UI 상태

새로고침 후에도 유지돼야 하는 사용자 설정.

ui:theme → light/dark/system ui:filters:{boardId} → 라벨/마감임박 필터 ui:collapsed-lists:{boardId} → 접힌 컬럼 목록

sessionStorage

탭 단위 임시 저장

실수로 탭을 닫아도 다시 열면 작성 중인 내용을 복구.

draft:card:{cardId} → 카드 모달 작성 중 description * 저장 시 자동 삭제 * 복원 시 "되돌리기" 제공

IndexedDB

오프라인 변경 큐

네트워크 끊긴 동안 변경을 적재 → 복귀 시 순서대로 재전송.

table pendingOps { id, type, method, path, body, createdAt, retries } 복귀 → flush(): MAX 3회 재시도 409 → board:sync 트리거
요건의 3배. 3종 모두 각자의 명확한 용도가 있어 단순 충족이 아닌 실용적 활용으로 차별화.
